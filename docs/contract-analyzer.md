# Анализатор договоров (`apps/contract-worker/contract_worker/analyzer.py`)

## Назначение
- Дать быстрый автоматический первичный скрининг текста договора.
- Вернуть структурированный JSON, пригодный для API, UI и последующей ручной проверки юристом.
- Не заменяет юридическое заключение, а приоритизирует задачи по уровню риска.

## Вход
- `document_text: str` — полный текст документа (сырой текст, может содержать переносы строк и лишние пробелы).

## Выход
`analyze_contract(...)` возвращает словарь с полями:
- `summary` — краткая сводка (слова, число риск-маркеров, итоговый уровень/балл).
- `risk_level` — `low | medium | high`.
- `risk_score` — целое `0..100`.
- `word_count` — число слов в документе.
- `risk_hits_total` — общее число срабатываний риск-маркеров.
- `risk_counts` — объект `{keyword: count}` по найденным маркерам.
- `risks` — отсортированный список уникальных найденных маркеров.
- `risk_snippets` — массив фрагментов текста вида `{keyword, snippet}` (ограничен по размеру).
- `top_words` — топ частотных слов вида `{word, count}`.

## Логика работы
1. Нормализация текста:
- схлопываются повторные пробелы и переносы строк;
- формируется lowercase-версия для поиска.

2. Токенизация:
- слова извлекаются regex-шаблоном `"[0-9A-Za-zА-Яа-яЁё_]+"`.

3. Поиск риск-маркеров:
- используется словарь `RISK_KEYWORDS` (английские и русские ключи);
- для каждого маркера считается число вхождений;
- дополнительно собираются контекстные сниппеты вокруг срабатываний (до лимита).

4. Расчет риска:
- `risk_hits_total = sum(risk_counts.values())`;
- `risk_density_per_1000_words = int((risk_hits_total / max(word_count, 1)) * 1000)`;
- `risk_score = min(100, risk_hits_total * 12 + min(40, risk_density_per_1000_words))`;
- уровни:
  - `high` при `risk_score >= 60`
  - `medium` при `risk_score >= 30`
  - `low` иначе

## Ограничения текущей версии
- Это rule-based эвристика без семантического разбора оговорок и исключений.
- Возможны ложные срабатывания по подстрокам.
- Морфология/синонимы покрыты ограниченно.
- Выход предназначен для triage, а не для финального правового вывода.

## Как расширять безопасно
- Добавлять новые маркеры в `RISK_KEYWORDS` только после проверки на ложные срабатывания.
- Изменения формулы `risk_score` проводить вместе с обновлением unit-тестов.
- При добавлении новых полей поддерживать обратную совместимость `result_json` для `core-api`.
