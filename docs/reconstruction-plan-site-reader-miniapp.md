# План капитальной реконструкции по ТЗ сайта, reader-бота и mini-app

Источник требований: `codex_unified_TZ_site_reader_miniapp.md`.

## 1. Цель и рамки

Цель реконструкции:
- собрать три согласованных интерфейса одной платформы (сайт, reader-бот, mini-app),
- встроить `Contract_AI_System` как центральный product-entry,
- развести аудитории `Юристы` и `Бизнес` во всех каналах,
- унифицировать CTA-framework: `Узнать` -> `Проверить/попробовать` -> `Решить/внедрить`.

Включено в scope:
- `apps/web` (сайт + mini-app контур),
- `apps/news/legacy/app/reader_bot.py` и связанные модули reader,
- интеграционные API в `apps/core-api` (reader, profile, saved/history, deep-link/flow),
- UX-маршруты между каналами.

Вне scope этого плана:
- глубокая переработка lead-бота,
- переработка contract-worker аналитики,
- редизайн контент-генератора вне нужного для reader/mini-app интерфейсного контракта.

## 2. Архитектурные решения перед стартом

1. Одна web-платформа на базе `apps/web`:
- сайт и mini-app живут в одном Next.js приложении (разные route-группы),
- это снижает потребление памяти и упрощает деплой на 2 CPU / 4 GB.

2. Reader-бот остается отдельным python-сервисом:
- роль: быстрый вход и маршрутизация,
- тяжелый UI и персональный кабинет уводим в mini-app.

3. `core-api` остается единым backend-контуром данных:
- источник правды для профиля, сохраненного, истории, анализов, CTA-событий.

## 3. Этапы реализации

## Этап 0. Product Contract (единый UX-контракт)

Результат:
- утвержден словарь терминов (одинаковый в 3 каналах),
- согласован IA-каркас сайта, reader-бота, mini-app,
- согласован единый CTA-framework по всем экранам,
- согласованы deep-link маршруты между каналами.

Задачи:
- зафиксировать route-map:
  - сайт: `Главная`, `Для юристов`, `Для бизнеса`, `Contract_AI_System`, `Решения/Услуги`, `Контент/Кейсы`, `О платформе/Контакты`;
  - reader: `Узнать`, `Проверить`, `Решения`, `Поиск`, `Мое`;
  - mini-app: `Главная`, `Контент`, `Инструменты`, `Решения`, `Мое`.
- утвердить шаблоны CTA для всех каналов.

Критерий выхода:
- есть единая матрица экранов и переходов; у каждого экрана есть цель и следующий шаг.

## Этап 1. Реконструкция IA и каркаса сайта

Результат:
- в `apps/web` реализован верхнеуровневый каркас по ТЗ,
- выделены явные входы `Для юристов` и `Для бизнеса`.

Задачи:
- перестроить верхнее меню и route-структуру,
- собрать обязательные блоки главной,
- создать/обновить страницы:
  - `/for-lawyers`,
  - `/for-business`,
  - `/contract-ai-system`,
  - `/solutions`,
  - `/content-cases`.
- вшить в каждую страницу понятные product/service CTA.

Критерий выхода:
- user flow сайта A/B/C/D из ТЗ проходит без тупиков.

## Этап 2. Product-entry и сервисные маршруты сайта

Результат:
- `Contract_AI_System` визуально и сценарно главный вход,
- сервисные сценарии не конкурируют с продуктом, а дополняют его.

Задачи:
- усилить product-лендинг `Contract_AI_System` (value -> задачи -> сценарии -> demo -> интеграции),
- на `Решения/Услуги` развести маршруты для юристов и бизнеса,
- на `Контент/Кейсы` внедрить контекстные CTA на продукт/решения/консультацию.

Критерий выхода:
- на любой странице есть понятный следующий шаг из 3 CTA-классов.

## Этап 3. Reader-бот: капитальная реконструкция меню и flow

Результат:
- reader-бот стал сценарным, а не "просто лентой",
- бот стабильно ведет в mini-app и product-flow.

Задачи:
- переделать старт и главное меню в структуру `Узнать / Проверить / Решения / Поиск / Мое`,
- реализовать обязательные user-flow 1..5 из ТЗ,
- в `Проверить` оставить короткий и прямой вход в `Contract_AI_System`,
- в `Поиск` добавить ключевые слова + тематические фильтры + переход в mini-app,
- в `Мое` дать персональный слой (saved/history/analyses/profile/interests/settings), даже если часть пока placeholder.

Критерий выхода:
- новый пользователь за 2-3 шага доходит либо до контента, либо до проверки договора, либо до решений.

## Этап 4. Mini-app: рабочая среда внутри `apps/web`

Результат:
- mini-app имеет полноценный 5-вкладочный рабочий контур,
- реализован онбординг и персонализация.

Задачи:
- реализовать нижнюю навигацию `Главная / Контент / Инструменты / Решения / Мое`,
- собрать экраны и блоки по ТЗ:
  - `Главная`: важное сегодня, для вас, польза, быстрый вход в Contract AI, продолжить/анализы, релевантные решения,
  - `Контент`: лента, фильтры, поиск, карточки,
  - `Инструменты`: Contract AI + история + место под будущие инструменты,
  - `Решения`: сценарии для юристов и бизнеса,
  - `Мое`: continuity-слой.
- реализовать онбординг с обязательными вопросами и влиянием на контент/CTA.

Критерий выхода:
- mini-app воспринимается как рабочее место, а не как дублирующая лента.

## Этап 5. Интеграции и связка 3 контуров

Результат:
- переходы между сайтом, reader-ботом и mini-app бесшовные,
- терминология и CTA синхронизированы.

Задачи:
- настроить и унифицировать deep-link:
  - сайт -> reader/mini-app,
  - reader -> mini-app/product,
  - mini-app -> решения/консультация/сайт.
- внедрить одинаковые UTM/event-параметры,
- нормализовать callback/data-contract между ботом и web.

Критерий выхода:
- любой ключевой сценарий можно продолжить в другом канале без потери контекста.

## Этап 6. Данные, API и персональный слой

Результат:
- API поддерживает персональные сценарии `saved/history/analyses/profile`,
- bot и mini-app работают на одном профиле пользователя.

Задачи:
- расширить/стабилизировать `core-api` reader endpoints,
- добавить недостающие поля профиля и интересов для персонализации,
- обеспечить согласованность `telegram_user_id` и профилей во всех каналах,
- добавить endpoint/сервис для выдачи "continue state".

Критерий выхода:
- пользователь видит одинаковое состояние в боте и mini-app.

## Этап 7. Нагрузочный и UX-hardening

Результат:
- контуры работают стабильно на целевом ресурсе сервера,
- устранены UX-тупики, лаги, дубли CTA и неочевидные переходы.

Задачи:
- профилировать `apps/web` и reader-бот,
- уменьшить лишние запросы в reader и mini-app,
- оптимизировать кэширование контента и профильных данных,
- ввести защиту от деградации при недоступности внешних API.

Критерий выхода:
- приемлемая скорость отклика и предсказуемый UX во всех ключевых flow.

## Этап 8. Релиз-контроль и приемка по ТЗ

Результат:
- формальная приемка по разделу 8 исходного ТЗ,
- готовый чек-лист Go/No-Go.

Задачи:
- провести проверку критериев:
  - сайт,
  - reader,
  - mini-app,
  - платформа в целом.
- провести smoke-тесты межканальных flow,
- зафиксировать rollback-план на случай проблем в проде.

Критерий выхода:
- все критерии готовности ТЗ закрыты и документально подтверждены.

## 4. Технический порядок запуска работ (практический)

1. Сначала `Этап 0` и `Этап 1` (без них дальше будет переписывание).
2. Затем параллельно:
- ветка `reader` (Этап 3),
- ветка `mini-app` (Этап 4),
- с общим контрактом API (Этап 6).
3. После слияния функционала — `Этап 5`, `Этап 7`, `Этап 8`.

## 5. Что даёт этот план

- минимизирует хаотичные правки,
- убирает дубли между каналами,
- сразу строит платформу как единую систему,
- оставляет место для роста `Contract_AI_System` и сервисных направлений.
